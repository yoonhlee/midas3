### [의사결정 1]

- 결정해야 할 것: 어떤 KPI와 지표를 기준으로 분석·대시보드를 만들 것인가 [D1][D2][D4][D7]
- 자주 발생하는 맥락: 팀이나 리더십이 성과 확인, 서비스 개선, 데이터 기반 의사결정을 위해 지표·대시보드·보고서를 요청할 때 [D2][D4][D7]
- 고려해야 할 요소 3~5개:
  - 비즈니스 목표와 현재 서비스 문제 [D2][D4]
  - 실제 추출 가능한 테이블과 데이터 범위 [D7]
  - 지표 정의의 일관성과 재사용 가능성 [D4][D7]
  - 이해관계자가 바로 이해할 수 있는 시각화·보고 형태 [D3][D7]
  - 지표가 이후 실험이나 실행으로 연결될 가능성 [D1][D5]
- 정답이 없는 이유: 같은 데이터라도 매출, 전환, 리텐션, 고객 행동 중 무엇을 우선 지표로 볼지에 따라 결론과 실행 방향이 달라진다 [D1][D2][D4]
- 좋은 결정 vs 나쁜 결정의 갈림길: 좋은 결정은 비즈니스 질문과 지표 정의가 맞고, 나쁜 결정은 추출 가능한 숫자만 나열해 의사결정으로 이어지지 않는다 [D2][D4][D7]

### [의사결정 2]

- 결정해야 할 것: 어떤 가설을 실험 또는 A/B 테스트로 검증할 것인가 [D1][D2][D5][D6]
- 자주 발생하는 맥락: 기능, 정책, 가격, 쿠폰, 마케팅 액션, 추천 모델의 효과를 확인해야 할 때 [D2][D5][D6]
- 고려해야 할 요소 3~5개:
  - 검증하려는 가설이 명확한지 [D1][D6]
  - 대상 고객군과 비교군을 어떻게 나눌지 [D2][D5]
  - 실험의 핵심 지표와 보조 지표를 무엇으로 둘지 [D2][D5]
  - 데이터 수집 이벤트가 정확히 설계·수집되는지 [D3]
  - 서비스 오류나 예외 상황을 모니터링할 수 있는지 [D6]
- 정답이 없는 이유: 어떤 실험이 좋은지는 서비스 목표, 데이터 품질, 실험 가능 기간, 고객 영향 범위에 따라 달라진다 [D2][D3][D6]
- 좋은 결정 vs 나쁜 결정의 갈림길: 좋은 결정은 검증 가능한 가설·지표·대상군을 명확히 하고, 나쁜 결정은 막연한 개선안을 실행한 뒤 결과 해석이 어렵게 만든다 [D1][D5][D6]

### [의사결정 3]

- 결정해야 할 것: 데이터 요청을 어디까지 추출·가공해 전달할 것인가 [D3][D7]
- 자주 발생하는 맥락: 다른 팀이나 담당자가 주문, 방문자 수, 성과, 주요 지표를 요청할 때 [D7]
- 고려해야 할 요소 3~5개:
  - 요청자의 실제 의도와 필요한 의사결정 [D3][D7]
  - 사용할 테이블과 필터 조건 [D7]
  - 데이터 정합성 확인 여부 [D3]
  - 단순 수치, 보고서, 대시보드 중 적절한 전달 형태 [D3][D7]
  - 반복 요청이면 자동화 또는 대시보드화할 필요가 있는지 [D3][D7]
- 정답이 없는 이유: 동일한 요청도 “수치만 필요한지”, “원인 분석이 필요한지”, “계속 볼 지표인지”에 따라 산출물이 달라진다 [D3][D7]
- 좋은 결정 vs 나쁜 결정의 갈림길: 좋은 결정은 요청 의도를 확인하고 필요한 수준까지 가공하지만, 나쁜 결정은 불완전한 숫자만 넘겨 추가 질문과 오류를 만든다 [D3][D7]

### [의사결정 4]

- 결정해야 할 것: 분석 결과를 어떤 실행안 또는 서비스 개선안으로 연결할 것인가 [D4][D5][D6]
- 자주 발생하는 맥락: 고객 행동 분석, 마케팅 성과 분석, 추천 모델, 가격예측, 서비스 개선 인사이트가 나왔을 때 [D4][D5][D6]
- 고려해야 할 요소 3~5개:
  - 분석 결과가 실제 비즈니스 문제와 연결되는지 [D4][D5]
  - 유관 부서가 실행할 수 있는 형태인지 [D4][D5]
  - 데이터의 한계와 예외 케이스가 확인됐는지 [D6]
  - 실행 후 모니터링 지표가 있는지 [D6]
  - 대시보드·보고서로 공유 가능한지 [D4][D7]
- 정답이 없는 이유: 분석 결과만으로는 실행 우선순위가 자동으로 정해지지 않고, 조직 목표·리소스·위험을 함께 판단해야 한다 [D4][D5][D6]
- 좋은 결정 vs 나쁜 결정의 갈림길: 좋은 결정은 데이터 근거와 실행 가능성을 함께 보고, 나쁜 결정은 통계적 차이나 상관만 보고 과한 결론을 낸다 [D5][D6]

- Cohort 분석: 같은 조건의 사용자 집단을 묶어 시간에 따른 행동을 비교하는 분석 방식 [D1][D4][D5]
- AARRR: 유입, 활성화, 유지, 매출, 추천 등 서비스 성장 지표를 보는 프레임 [D1]
- A/B 테스트: 두 가지 이상 안을 비교해 효과를 검증하는 실험 방식 [D2][D4][D5]
- 데이터 정합성: 데이터가 의도한 기준과 맞게 수집·관리되는지 확인하는 품질 기준 [D3]
- RFM / CLV: 고객 구매 행동과 고객 생애가치를 기준으로 세그먼트를 나누는 분석 방식 [D5]

## [Part 4. 협업과 커뮤니케이션] ← 시나리오의 사회적 맥락

**자주 함께 일하는 직군:**  
※ 채용공고 협업 부서 + 인터뷰의 협업 언급  
※ 직군별로 “어떤 관계인가, 무엇을 주고받는가”

- PM / PD / 엔지니어: 제품·서비스 문제를 함께 정의하고, 분석 결과를 제품 의사결정에 반영함 [D4]
- 스쿼드·리더십: BI 리포트와 대시보드를 통해 핵심 지표와 의사결정 자료를 공유받음 [D4]
- 데이터플랫폼팀: 데이터 거버넌스와 데이터 리터러시 강화를 위해 협업함 [D4]
- 데이터 엔지니어: 적재·관리한 데이터를 분석가가 받아 비즈니스 인사이트나 추천·예측에 활용함 [D6]
- 마케터·데이터 엔지니어링/플랫폼팀: 마케팅 데이터 분석, 데이터 사용 환경, 성과 분석을 위해 협업함 [D5]
- 유관 부서 요청자: 요청 의도를 확인하고 필요한 데이터를 추출·가공해 전달함 [D3]

**자주 받는 요청의 형태 3~5개:**  
※ 직무 인터뷰/회고록에서 “이런 요청을 받는다” 언급 추출  
※ 따옴표 포함 직접 인용 형태  
※ 누가 보내는지도 명시

1. (보내는 사람: 데이터팀 담당자) “주문 관련 테이블은 앞으로 자주 볼 거니까...” [D7]
2. (보내는 사람: 실무 요청자) “일일 방문자 수 좀 뽑아서...” [D7]
3. (보내는 사람: 실무 요청자) “성과 분석 보고서 간단히 작성해주세요.” [D7]
4. (보내는 사람: 실무 요청자) “필요한 지표들 정의해서 초안 만들어 주세요.” [D7]

**요청을 받았을 때의 전형적 응답 흐름:**  
※ 인터뷰에서 “이런 요청 받으면 어떻게 처리하는지” 언급 추출  
※ 미션의 “관찰→가설→제안” 구조와 매칭됨

단계 1: 요청자가 원하는 의사결정과 필요한 지표를 확인한다 [D3][D7]  
단계 2: 사용할 테이블, 이벤트, 로그, 데이터 범위를 확인한다 [D3][D7]  
단계 3: SQL/BigQuery/Python으로 데이터를 추출·전처리한다 [D1][D3][D7]  
단계 4: 데이터 정합성, 앱 이벤트 수집 정확도, 예외 케이스를 점검한다 [D3][D6]  
단계 5: KPI, Funnel, Cohort, A/B 테스트 결과 등으로 분석한다 [D1][D2][D4][D5]  
단계 6: 대시보드, BI 리포트, PPT/Word 보고서로 전달하고 실행·모니터링 방향을 논의한다 [D3][D4][D7]

**결과물의 전달 형태:**  
※ 보고서? 슬랙 메시지? 대시보드? 구두 보고?  
※ 미션의 “입력 형식” 설계에 사용

- SQL 쿼리와 추출 데이터: 반복 데이터 요청이나 지표 추출 요청에 사용 [D7]
- 이벤트 검증 결과·이슈 리포트: 앱 이벤트 수집 정확도 확인이나 데이터 정합성 이슈 공유에 사용 [D3]
- BI 리포트·대시보드: 핵심 지표 모니터링과 리더십·팀 의사결정에 사용 [D2][D3][D4][D7]
- PPT/Word 분석 보고서: 성과 분석, 지표 정의, 시각화 결과, 개선 제안을 문서로 전달할 때 사용 [D7]
- PDF/PPT/Word 사전 과제형 산출물: 채용 과제나 분석 결과 제출 상황에서 사용 [D3]
